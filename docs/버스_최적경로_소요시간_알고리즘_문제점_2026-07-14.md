# 버스 최적 경로·소요시간 알고리즘 문제점 및 개선 인수인계 — 2026-07-14

## 1. 문서 목적

이 문서는 두 장소 사이의 버스 최적 경로와 예상 소요시간을 계산하는 기능을 구현하기 전에, 현재 코드와 기존 설계에서 확인된 문제를 정리한 인수인계 문서다.

다음 작업자는 이 문서를 먼저 읽고 현재 설계 문서와 구현 계획을 그대로 구현하지 말고, 아래 문제를 반영해 설계를 갱신해야 한다.

관련 파일:

```text
app/transit.py
collectors/tago_bus_routes.py
scripts/build_route_stops.py
db/travel.db
docs/superpowers/specs/2026-07-14-bus-route-multi-transfer-design.md
docs/superpowers/plans/2026-07-14-bus-route-multi-transfer.md
docs/tago_md/TAGO_버스위치정보_API_가이드.md
```

## 2. 현재 구현 상태

현재 `main` 브랜치의 `app/transit.py`는 예상 소요시간을 계산하지 않는다.

현재 지원 범위:

- 출발·도착 좌표 주변 정류소 검색
- 같은 노선·방향의 직행 경로 탐색
- 동일 `stop_id`에서 갈아타는 1회 환승 경로 탐색
- 직행 경로의 정류장 수 계산

현재 지원하지 않는 범위:

- 장소명에서 좌표 변환을 포함한 공개 경로 추천 함수
- 2~3회 환승
- 출발지·목적지 및 환승 도보시간
- 버스 대기시간
- 버스 승차시간
- 총 예상 소요시간
- 실시간 도착정보
- 실시간 차량 위치
- 경로별 신뢰도 및 추정 여부

따라서 현재 상태는 `경로 연결성 탐색의 초기 구현`이며 `최적 소요시간 계산기`는 아직 구현 전이다.

## 3. 현재 버스 데이터 상태

버스 데이터는 TAGO 단일 원천으로 정리되어 있다.

| 테이블 | 건수 | 식별자 |
|---|---:|---|
| `transport` | 3,061 | TAGO `nodeId` |
| `bus_route` | 128 | TAGO `routeId` |
| `bus_route_stop` | 9,874 | `routeId + updowncd + nodeord` |

검증 결과:

- TAGO 원천 대비 누락 노선 0개
- TAGO 원천 대비 누락 정류소 0개
- TAGO 원천 대비 누락 경유 순서 0개
- 정류소명·좌표 불일치 0개
- 외래키 오류 0개
- SQLite 무결성 `ok`

정적 경로 데이터는 준비되어 있으므로 다음 단계의 핵심은 탐색 및 시간 계산 알고리즘이다.

## 4. 확인된 핵심 문제

### 4.1 출발·도착 정류소를 가장 가까운 2개로 제한

현재 코드는 출발지와 도착지에서 각각 거리순 정류소 2개만 사용한다.

문제:

- 가장 가까운 정류소가 목적지 방향 승강장이 아닐 수 있음
- 같은 정류소명도 방향별 `nodeId`가 다름
- 필요한 승강장이 거리순 3번째 이후일 수 있음
- 정상적인 직행 경로를 `no route`로 판단할 수 있음

실제 확인 사례:

```text
성심당 → 갤러리아 타임월드점
```

- 정류소 후보 2개 사용: 경로 없음
- 후보를 10개로 확대: 직행 경로 발견

개선 방향:

```text
일정 반경 내 정류소 전체 조회
→ 실제 노선 연결 가능성 확인
→ 도보거리와 전체 경로 비용으로 후보 평가
```

단순히 `k=2`를 `k=3`으로 바꾸는 것으로 해결하지 않는다.

### 4.2 동일한 `stop_id`에서만 환승 가능

현재 환승 탐색은 첫 번째 노선의 하차 정류소와 두 번째 노선의 승차 정류소가 동일한 `nodeId`일 때만 가능하다.

현실의 환승 형태:

- 길 건너편 정류소로 이동
- 사거리의 다른 모퉁이로 이동
- 같은 정류소명이지만 방향별 `nodeId`가 다름
- 약 50~250m 떨어진 정류소로 도보 이동

개선 방향:

```text
버스 간선:
  같은 routeId + updowncd에서 뒤쪽 nodeord로 이동

도보 환승 간선:
  서로 다른 nodeId지만 허용 반경 이내인 정류소로 이동
```

환승 도보 반경은 일괄 확정하지 말고 100m, 150m, 250m 등의 회귀 테스트로 결정한다.

### 4.3 승차·하차 정류장의 다음 버스 도착시간 차이 사용 금지

기존 설계는 다음 계산을 고려한다.

```text
하차 정류장의 다음 버스 도착예정시간
- 승차 정류장의 다음 버스 도착예정시간
= 버스 승차시간
```

이 방식은 사용하면 안 된다.

이유:

- 두 응답의 `다음 버스`가 같은 차량이라는 보장이 없음
- 앞 차량과 뒤 차량을 비교할 수 있음
- 결과가 음수가 될 수 있음
- 배차간격만큼 과대 또는 과소 계산될 수 있음

동일 차량번호 또는 동일 차량 식별자를 확실히 연결할 수 있을 때만 차량 위치 기반 이동시간 보정을 허용한다.

### 4.4 환승 대기시간을 현재 시각 기준으로 조회

환승 정류소의 도착정보를 지금 조회해도 사용자는 첫 번째 버스를 타고 수십 분 뒤에 해당 정류소에 도착한다.

예:

```text
현재 시각: 14:00
환승 버스 도착: 14:05
사용자의 환승 정류소 도착 예상: 14:20
```

14:05 버스는 사용자가 도착하기 전에 지나가므로 대기시간 5분으로 계산하면 안 된다.

환승 대기는 다음 시점을 기준으로 계산해야 한다.

```text
첫 번째 버스 하차 예상 시각
+ 환승 도보시간
→ 환승 승차 가능 시각
```

이 미래 시각 이후의 정확한 도착정보를 구할 수 없으면 평균 배차간격 또는 별도 추정치로 처리하고 `estimated=true`를 표시한다.

### 4.5 정류소 간 거리와 고정 평균속도로 승차시간 계산

정류소 좌표를 순서대로 연결한 거리를 시속 20km로 나누는 방식은 기본 근사치로는 사용할 수 있지만 정확한 실시간 시간으로 표시하면 안 된다.

반영되지 않는 요소:

- 실제 도로 굴곡
- 신호 대기
- 정류소 승하차 시간
- 시간대별 교통량
- 버스전용차로
- 회차 및 유턴
- 공사·사고·임시 우회

권장:

- 정적 1차 후보 평가용으로만 사용
- 최종 결과에서는 추정값임을 표시
- 실제 관측 데이터를 확보하면 노선·구간·시간대별 보정계수를 적용

### 4.6 모든 노선에 고정 대기시간 5분 적용

노선과 시간대를 무시한 고정 대기시간은 외곽·야간 노선을 지나치게 유리하게 평가한다.

고려해야 할 요소:

- 노선별 배차간격
- 평일·주말·공휴일
- 출퇴근·낮·야간
- 첫차·막차
- 현재 운행 차량 유무

실시간 도착정보가 없으면 최소한 결과에 추정 대기시간과 근거를 표시해야 한다.

### 4.7 도보시간을 직선거리로 계산

직선거리 기반 도보시간에는 다음 요소가 없다.

- 횡단보도 및 신호 대기
- 육교·지하도
- 건물 출입구
- 담장·하천·철도
- 보행 불가능 도로
- 경사도

특히 도로 반대편 정류소는 좌표상 가깝지만 실제 이동시간이 더 길 수 있다.

MVP에서는 직선거리 추정을 사용할 수 있으나 반드시 추정값으로 처리하고, 이후 보행 경로 API 또는 도로 네트워크를 연동할 수 있게 함수 경계를 분리한다.

### 4.8 목적지와 계속 가까워지는 정류소만 탐색

기존 다중 환승 설계의 `목적지 직선거리가 감소하는 정류소만 확장` 규칙은 좋은 경로를 제거할 수 있다.

예:

```text
출발지
→ 목적지와 잠시 멀어지는 대형 환승 거점
→ 급행버스로 목적지 이동
```

각 구간이 목적지에 가까워져야 한다는 조건을 필수 조건으로 사용하지 않는다. 거리 감소는 탐색 우선순위나 휴리스틱으로만 사용한다.

### 4.9 단계별 거리순 상위 200개 Beam 제한

거리순 후보 200개 제한은 탐색 속도를 높이지만 다음 좋은 경로를 조기에 제거할 수 있다.

- 곧 도착하는 버스
- 환승이 적은 경로
- 급행버스 연결 경로
- 도보가 짧은 경로

Beam을 사용한다면 직선거리 하나가 아니라 복합 점수를 사용하고, 회귀 테스트에서 정답 경로가 유지되는지 검증해야 한다.

### 4.10 순환·회차 노선에서 동일 정류소 반복

실제 DB에는 같은 노선·방향에서 동일 정류소가 여러 `nodeord`에 나타나는 경우가 있다.

확인 사례:

```text
특구1 신세계백화점: 순번 14, 20
2번 봉산동기점: 순번 1, 48
```

경로 상태에 정류소 ID만 저장하면 어느 순번에서 승하차했는지 구분할 수 없다.

각 승차 구간은 최소한 다음 정보를 보존해야 한다.

```text
route_id
updowncd
board_stop_id
board_order
alight_stop_id
alight_order
```

### 4.11 노선 시퀀스만으로 중복 경로 제거

같은 노선 조합이어도 환승 정류소가 다르면 도보거리와 대기시간이 달라질 수 있다.

```text
705 → 911, A 정류소 환승
705 → 911, B 정류소 환승
```

노선 ID 시퀀스만으로 중복 제거하지 않는다. 승차·하차·환승 정류소를 포함한 경로 구조와 비용을 함께 비교한다.

### 4.12 실시간값과 추정값 혼합

경로마다 실시간 데이터 적용률이 다를 수 있다.

```text
경로 A: 전 구간 실시간
경로 B: 전 구간 추정
경로 C: 첫 구간 실시간 + 환승 구간 추정
```

이들을 동일한 정확도의 숫자처럼 정렬하고 표시하면 안 된다.

반환값에 다음 정보를 포함한다.

```json
{
  "total_minutes": 35,
  "estimated": true,
  "realtime_coverage": 0.5,
  "calculated_at": "ISO-8601 시각"
}
```

### 4.13 운행 종료·운휴·막차 처리 없음

정적 경로가 존재해도 현재 이용 가능한 노선이라는 보장은 없다.

구분해야 할 상태:

- 정상 운행
- 현재 운행 차량 없음
- 첫차 전
- 막차 종료
- 임시 운휴
- API 실패 또는 호출 제한

`실시간 응답 없음`을 바로 `운행 종료`로 판단하지 않는다. API 실패와 실제 무운행을 별도 상태로 표현한다.

### 4.14 실시간 API 호출량과 응답 지연

모든 후보의 모든 승·하차 정류소를 실시간 조회하면 호출량과 응답시간이 커진다.

권장:

- 정적 계산으로 후보를 먼저 축소
- 상위 10~15개 후보만 실시간 보정
- 동일 `routeId + nodeId` 요청은 한 요청 안에서 캐싱
- 실시간 위치는 10~20초 TTL 캐시 검토
- API 시간초과 시 전체 경로 추천을 실패시키지 않고 추정값으로 폴백

## 5. 권장 계산 구조

### 5.1 1단계 — 정적 후보 탐색

```text
장소명 → 좌표
→ 반경 내 출발·도착 정류소 후보
→ 버스 이동 간선 + 도보 환승 간선 그래프
→ 직행부터 최대 허용 환승까지 후보 탐색
```

탐색 상태에는 정류소뿐 아니라 현재 노선·방향·순번·사용한 노선을 포함한다.

### 5.2 2단계 — 정적 근사 점수

```text
출발 도보
+ 구간별 정적 승차시간
+ 추정 대기시간
+ 환승 도보
+ 환승 페널티
+ 목적지 도보
```

이 단계는 API 호출 없이 상위 후보를 고르는 용도다.

### 5.3 3단계 — 실시간 보정

상위 후보에만 다음을 적용한다.

- 첫 탑승 정류소의 도착예정시간
- 해당 노선의 현재 운행 차량 존재 여부
- 동일 차량을 확인할 수 있을 때만 위치 기반 승차시간 보정
- 실시간 데이터가 없으면 구간별 정적 추정값 유지

환승 대기는 미래의 환승 가능 시각을 기준으로 해야 하며, 정확한 미래 도착예측을 구할 수 없으면 추정값으로 표시한다.

### 5.4 4단계 — 최종 정렬 및 반환

최종 정렬 시 총시간 하나만 보지 말고 다음 요소를 함께 고려한다.

- 총 예상시간
- 환승 횟수
- 총 도보시간
- 최대 한 번의 도보 구간
- 실시간 데이터 적용률
- 추정값 비율
- 운행 가능 상태

## 7. 구현 우선순위

### P0 — 경로 정확성

1. 출발·도착 정류소 `k=2` 제한 제거
2. 도보 환승 간선 추가
3. 순환 노선의 `nodeord` 상태 보존
4. 노선 시퀀스만 사용하는 중복 제거 수정

### P1 — 시간 계산 정확성

1. 서로 다른 다음 버스 도착시간 차이 사용 금지
2. 첫 승차 대기시간만 우선 실시간 적용
3. 환승 미래 시각 문제 반영
4. 정적 승차시간과 실시간값의 출처 구분
5. 결과에 `estimated`, `realtime_coverage` 추가

### P2 — 운영 안정성

1. 실시간 API 타임아웃과 폴백
2. 요청 단위 및 TTL 캐시
3. 운행 종료와 API 실패 상태 분리
4. 호출량 제한 및 로깅

## 8. 필수 회귀 테스트

최소 다음 사례를 자동 테스트로 고정한다.

### 사례 A — 가까운 정류소 2개에 필요한 노선이 없는 경우

```text
성심당 → 갤러리아 타임월드점
```

거리순 소수 후보 때문에 경로가 사라지지 않아야 한다.

### 사례 B — 1회 환승

```text
성심당 → 대전시립박물관
```

환승 경로가 최소 하나 이상 반환되어야 한다.

### 사례 C — 도보 환승

하차 `nodeId`와 다음 승차 `nodeId`가 다르지만 허용 도보 반경 안에 있는 경로를 테스트한다.

### 사례 D — 순환 노선

동일 정류소가 서로 다른 `nodeord`에 반복되는 노선에서 올바른 승·하차 순번을 선택해야 한다.

### 사례 E — 실시간 API 실패

API가 시간초과 또는 오류를 반환해도 정적 추정 경로를 반환하고 `estimated=true`여야 한다.

### 사례 F — 잘못된 실시간 시간 차이

승차 정류장 ETA가 10분, 하차 정류장 ETA가 4분인 응답을 주입해도 `ride_minutes=-6` 같은 결과가 발생하면 안 된다.

## 9. 완료 조건

다음 조건을 모두 만족해야 작업 완료로 본다.

- 정류소 후보 수 하드코딩 때문에 알려진 경로가 누락되지 않음
- 다른 `nodeId` 사이의 도보 환승 지원
- 최대 허용 환승 횟수까지 탐색 가능
- 모든 leg에 노선·방향·승하차 순번이 명시됨
- 음수 대기시간 또는 승차시간이 발생하지 않음
- 실시간 API 실패 시 전체 요청이 실패하지 않음
- 실시간값과 추정값이 필드로 구분됨
- 환승 대기시간을 현재 시각의 단순 ETA로 잘못 합산하지 않음
- 필수 회귀 테스트가 자동화됨
- 실제 DB 3,061개 정류소와 9,874개 경유 데이터를 사용한 통합 테스트 통과

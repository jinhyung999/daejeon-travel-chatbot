# 음식점 추천 데이터 DB 반영 설계

## 목표

검토가 끝난 네이버 음식점 후보와 기존 추천 CSV를 SQLite `place` 테이블에 통합한다. 동일 음식점은 기존 행을 보강하고, DB에 없는 음식점만 새로 추가한다. 이후 LLM이 추천 음식점을 우선 조회할 수 있도록 `recommend='추천'` 값을 제공한다.

## 입력 데이터

- `outputs/restaurant-review/restaurant_candidates_review_완료_승인반영.xlsx`
  - 승인 상태의 기준이며 승인 438개, 제외 22개다.
- `data/curation/restaurant_candidates.csv`
  - 주소, 원본 URL, 좌표, 추천 점수와 사유의 원천이다.
- `data/curation/restaurant_recommendations.csv`
  - 기존 추천 263개
- `db/travel.db`
  - 현재 음식점 19,287개

사전 비교 결과 두 추천 원천에서 동일 음식점으로 판단되는 항목은 5개다. 승인 목록 내부에는 동일 음식점 중복이 없다. 두 원천을 논리적으로 합친 추천 음식점은 696개다.

## 스키마

`place` 테이블에는 다음 컬럼 하나만 추가한다.

```sql
recommend TEXT
```

- 추천 음식점: `추천`
- 일반 음식점: `NULL`
- 신규 DB를 위한 `db/schema.sql`과 기존 `travel.db` 마이그레이션을 모두 지원한다.
- 추천 음식점 조회를 위해 `category, recommend` 복합 인덱스를 추가한다.

위도와 경도는 기존 `lat`, `lng` 컬럼 하나만 사용한다. 네이버 전용 좌표 컬럼이나 블로그 통계 컬럼은 추가하지 않는다.

## 동일 음식점 판별

상호명은 Unicode NFKC 정규화 후 공백과 문장부호를 제거해 비교한다. 주소는 `대전광역시`와 `대전` 표기를 통일한다.

다음 중 하나를 만족하면 기존 DB 행과 동일 음식점으로 본다.

1. 정규화한 상호명이 같고 좌표 거리가 100m 이내다.
2. 상호명이 서로 포함되거나 문자열 유사도가 0.72 이상이며 좌표 거리가 50m 이내다.
3. 한쪽 좌표가 없을 때 정규화한 상호명과 주소가 모두 일치한다.

이름이 같더라도 다른 구에 있거나 거리가 기준을 넘으면 별도 지점으로 유지한다. 같은 후보가 여러 DB 행과 매칭되면 아래 순서로 대표 행 하나를 선택한다.

1. 기존 추천 CSV에 포함된 행
2. 데이터 출처 우선순위: `tourapi`, `daejeon_food`, `sbiz`, `naver_search`
3. 후보 좌표와의 거리
4. `place_id` 사전순

기존 DB의 중복 행을 삭제하거나 합치지는 않는다. 이번 작업에서는 선택한 대표 행만 추천으로 표시한다.

## 기존 행 보강

기존 DB 행과 매칭되면 다음과 같이 처리한다.

- `recommend='추천'`으로 설정한다.
- 기존 `lat`, `lng`가 모두 있으면 유지한다.
- 둘 중 하나라도 비어 있으면 좌표 쌍의 일관성을 위해 네이버 `lat`, `lng`를 함께 채운다.
- 기존 주소와 홈페이지가 비어 있을 때만 승인 데이터로 채운다.
- 기존 `extra_json` 객체를 보존하면서 `recommendation` 객체를 병합한다.

`extra_json.recommendation`에는 다음 정보만 저장한다.

- `source`: `naver_review`
- 네이버 상세 업종
- 추천 점수와 추천 사유
- 네이버 도로명 주소
- 네이버 업체 링크
- 네이버에서 확인한 위도와 경도

블로그 글 수, 블로거 수, 최근 글 날짜, 검색 결과 수는 저장하지 않는다.

키 이름은 다음 구조로 고정한다.

```json
{
  "recommendation": {
    "source": "naver_review",
    "detailed_category": "한식>칼국수",
    "score": 87,
    "reason": "지역성: 칼국수",
    "road_address": "대전광역시 ...",
    "naver_link": "https://...",
    "naver_latitude": 36.0,
    "naver_longitude": 127.0
  }
}
```

## 신규 행 추가

기존 DB와 매칭되지 않는 승인 음식점만 새 `place` 행으로 추가한다.

- `place_id`: 상호명, 도로명 주소, 좌표를 사용한 결정적 해시 ID
- `name`: 승인된 상호명
- `category`: `restaurant`
- `address`: 도로명 주소가 있으면 도로명 주소, 없으면 지번 주소
- `lat`, `lng`: 네이버 후보 좌표
- `source_api`: `naver_search`
- `homepage`: 네이버 업체 링크가 있으면 저장
- `recommend`: `추천`
- `extra_json`: 기존 행과 같은 `recommendation` 구조

같은 입력을 다시 반영해도 동일한 `place_id`와 매칭 규칙을 사용하므로 중복 삽입되지 않아야 한다.

## 데이터 흐름

1. 완료 엑셀에서 `approved`인 행 번호를 추출하고, 같은 행의 원본 값은 `restaurant_candidates.csv`에서 읽는다. 상호명과 주소도 함께 대조해 행 순서 불일치를 차단한다.
2. 기존 추천 CSV의 `place_id` 263개를 추천으로 표시한다.
3. 승인 후보를 전체 음식점 DB와 비교한다.
4. 동일 음식점이면 기존 대표 행을 보강하고 추천으로 표시한다.
5. 매칭되지 않으면 신규 추천 음식점으로 삽입한다.
6. 모든 변경을 단일 SQLite 트랜잭션으로 커밋한다.
7. 결과 수와 중복, 좌표, JSON 유효성을 검증한다.

사전 분석 기준으로 승인 후보 438개 중 약 331개는 기존 DB 행을 재사용하고 약 107개가 신규 삽입 대상이다. 최종 수치는 구현한 동일 음식점 판별 결과를 검증해 확정한다.

## 안전성과 오류 처리

- 변경 전에 `travel.db` 백업본을 만든다.
- 스키마 마이그레이션과 데이터 반영은 재실행 가능해야 한다.
- 엑셀 상태가 `approved`가 아닌 행은 반영하지 않는다.
- 좌표가 대전 범위를 벗어나거나 숫자가 아니면 신규 삽입하지 않고 오류로 보고한다.
- `extra_json`이 객체가 아니거나 손상된 기존 행은 원본 문자열을 잃지 않도록 오류 목록에 기록하고 해당 행 변경을 중단한다.
- 전체 반영 도중 오류가 발생하면 트랜잭션을 롤백한다.

## 검증 및 테스트

테스트는 구현보다 먼저 작성한다.

- 마이그레이션을 두 번 실행해도 `recommend` 컬럼과 인덱스가 중복 생성되지 않는다.
- 기존 추천 CSV의 모든 `place_id`가 `recommend='추천'`이 된다.
- 동일 상호명·근거리 후보는 기존 행을 재사용한다.
- 유사 상호명·50m 이내 후보는 기존 행을 재사용한다.
- 같은 이름이라도 다른 구 또는 원거리 지점은 별도 행으로 추가한다.
- 기존 좌표가 있으면 바꾸지 않고, 없을 때만 네이버 좌표를 채운다.
- 기존 `extra_json` 키를 보존하면서 추천 정보만 병합한다.
- 블로그 통계 필드는 DB와 `extra_json`에 저장되지 않는다.
- 재실행해도 행 수와 추천 수가 변하지 않는다.
- 반영 후 추천 음식점은 논리적 중복을 제외해 696개여야 한다.
- `PRAGMA integrity_check` 결과가 `ok`이고 모든 추천 행의 JSON이 유효해야 한다.

## LLM 연동 경계

이번 작업은 추천 데이터의 DB 기반만 만든다. 현재 프로젝트에는 음식점 LLM 추천 쿼리가 구현되어 있지 않으므로 LLM 호출 코드는 변경하지 않는다. 이후 추천 후보 조회 시 다음과 같이 우선순위를 적용할 수 있다.

```sql
ORDER BY CASE WHEN recommend = '추천' THEN 0 ELSE 1 END
```

필요하면 `WHERE recommend='추천'`으로 검토된 추천 음식점만 LLM 컨텍스트에 전달할 수 있다.

# 네이버 기반 대전 음식점 후보 수집 설계

## 목적

기존 `data/curation/restaurant_recommendations.csv`의 263개 음식점과 겹치지 않는 신규 후보를 대덕구, 유성구, 동구, 서구, 중구별로 80~100개 수집한다. 네이버 지역검색 API로 후보를 찾고 네이버 블로그 검색 API의 검색 메타데이터로 추천 가능성을 보강한다. 결과는 사람이 표 형태로 검수할 수 있는 별도 CSV로 만든다.

최종 추천 목록 반영은 이번 범위에 포함하지 않는다. 이번 작업의 완료 조건은 중복과 명백한 부적합 후보가 정리된 검수용 후보 CSV와 재실행 가능한 수집기를 제공하는 것이다.

## 선택한 접근법

지역검색과 블로그 검색을 결합하는 반자동 방식을 사용한다.

- 지역검색은 식당명, 분류, 주소, 좌표와 검색 노출 신호를 제공한다.
- 블로그 검색은 본문을 수집하지 않고 검색 결과 수, 최근 게시물 수, 작성자 다양성, 최신 작성일만 파생한다.
- 자동 점수는 후보의 검토 순서를 정할 뿐 최종 추천을 확정하지 않는다.
- 검수자가 `review_status`를 `approved` 또는 `rejected`로 변경해야 최종 추천으로 사용할 수 있다.

지역검색만 사용하는 방식은 추천 근거가 부족하고, 공공데이터만 사용한 뒤 네이버로 검증하는 방식은 신규 후보 발견 범위가 좁어 채택하지 않는다.

## 파일과 구성 요소

### 수집기

`scripts/collect_naver_restaurant_candidates.py`를 추가한다.

책임은 다음과 같이 분리한다.

1. 네이버 API 클라이언트: 인증 헤더, 요청 제한, 재시도, 응답 검증을 담당한다.
2. 검색어 생성기: 구·동네·음식 키워드 조합을 결정적인 순서로 생성한다.
3. 후보 정규화기: HTML 태그, 공백, 문장부호와 지점 표기를 정리한다.
4. 후보 필터: 대전 주소, 대상 구, 음식점 분류와 여행 추천 적합성을 판정한다.
5. 중복 판정기: 기존 추천 CSV, 기존 DB 좌표, 같은 실행에서 수집된 후보를 비교한다.
6. 블로그 보강기: 검색 결과 메타데이터를 집계해 후보별 지표를 계산한다.
7. 점수 계산기: 지역검색 신호, 블로그 지표, 지역성 키워드와 감점 사유를 합산한다.
8. CSV 내보내기: 구별 상위 80~100개를 안정된 정렬 순서로 기록한다.

구성값과 키워드 목록은 같은 모듈의 상수로 시작한다. 별도 설정 파일은 사용자가 반복적으로 목록을 조정할 필요가 생길 때만 분리한다.

### 출력 파일

기본 출력은 `data/curation/restaurant_candidates.csv`이다. UTF-8 BOM 없이 RFC 4180 형식으로 작성하고, 모든 텍스트 필드는 한 줄로 정규화해 VS Code 표 편집기의 가독성을 유지한다.

열 순서는 다음과 같다.

| 열 | 의미 |
|---|---|
| `review_status` | 기본값 `pending`; 검수자가 `approved` 또는 `rejected`로 변경 |
| `district` | 대덕구, 유성구, 동구, 서구, 중구 중 하나 |
| `name` | HTML 표시 태그를 제거한 식당명 |
| `category` | 네이버 지역검색 분류 |
| `address` | 지번 주소 |
| `road_address` | 도로명 주소 |
| `latitude`, `longitude` | WGS84 좌표 |
| `naver_link` | 지역검색 응답의 상세 링크. 값이 없으면 빈 문자열 |
| `blog_search_url` | 검수자가 브라우저에서 열 수 있는 식당명·지역 블로그 검색 URL |
| `matched_queries` | 후보가 발견된 검색어를 ` | `로 연결한 값 |
| `local_hit_count` | 서로 다른 지역검색 조합에서 발견된 횟수 |
| `comment_sort_hit_count` | 리뷰 개수순 검색에서 발견된 횟수 |
| `blog_result_count` | 식당명과 지역을 결합한 블로그 검색의 `total` 값 |
| `recent_blog_count` | 날짜순 상위 결과 중 최근 12개월 게시물 수 |
| `distinct_blogger_count` | 정확도순 상위 결과의 서로 다른 블로그 수 |
| `latest_post_date` | 날짜순 결과의 최신 작성일 |
| `recommendation_score` | 검토 우선순위를 위한 내부 점수 |
| `recommendation_reason` | 지역성, 대표 음식, 최근 언급 등 짧은 자동 근거 |
| `possible_duplicate` | 자동 확정할 수 없는 기존 목록 유사 후보 표시 |
| `reject_reason` | 자동 제외 사유 또는 검수자가 입력할 사유 |

블로그 제목, 본문 요약, 본문, 이미지와 작성자 개인정보는 CSV에 저장하지 않는다.

## 검색 전략

지역검색 API가 요청당 최대 5건만 반환하므로 하나의 광범위한 검색어에 의존하지 않는다.

### 위치 시드

- 대덕구: 대덕구, 신탄진, 송촌동, 비래동, 오정동, 중리동
- 유성구: 유성구, 봉명동, 궁동, 어은동, 관평동, 전민동, 노은동, 지족동, 원내동
- 동구: 동구, 대전역, 소제동, 가양동, 용운동, 판암동, 산내
- 서구: 서구, 둔산동, 갈마동, 월평동, 도마동, 관저동, 만년동, 탄방동
- 중구: 중구, 대흥동, 은행동, 선화동, 오류동, 유천동, 산성동, 보문산

### 음식 시드

`맛집`, `한식`, `향토음식`, `노포`, `칼국수`, `국밥`, `냉면`, `두부두루치기`, `삼계탕`, `고기`, `해산물`, `중식`, `일식`, `분식`을 사용한다.

각 위치·음식 조합을 `sort=comment`와 `sort=random`으로 각각 호출한다. 네이버 명세에서 `random`은 기본 정확도 정렬에 해당한다. 한 구에서 필터 통과 고유 후보가 120개에 도달하면 추가 지역검색을 중단한다. 전체 조합이 끝나도 80개 미만이면 해당 구 결과를 그대로 내보내고 부족 수량을 종료 요약에 명시한다. 결과를 임의 복제하거나 품질 기준을 낮춰 80개를 채우지 않는다.

블로그 검색은 지역 필터를 통과하고 기존 목록의 확정 중복이 아닌 후보에만 수행한다. 후보마다 다음 두 요청을 사용한다.

- 정확도순 최대 100건: `blog_result_count`, `distinct_blogger_count` 계산
- 날짜순 최대 100건: `recent_blog_count`, `latest_post_date` 계산

검색어는 `식당명 + 도로명 주소의 동/구`를 사용해 동명이인 식당의 영향을 줄인다.

지역검색의 `mapx`, `mapy` 정수는 WGS84 경도·위도에 10,000,000을 곱한 값으로 해석해 각각 `longitude`, `latitude` 실수로 변환한다. 최근 12개월은 실행일을 기준으로 계산한다.

## 필터와 프랜차이즈 정책

다음 항목만 즉시 제외한다.

- 주소가 대전광역시가 아니거나 대상 구와 일치하지 않음
- 음식점이 아닌 기관, 소매점, 숙박업소
- 일반 카페, 디저트, 베이커리만을 주업으로 하는 분류
- 설명이나 분류에 배달전문 또는 포장전문이 명시됨

치킨, 피자, 햄버거, 도시락, 패스트푸드와 전국 단위 일반 프랜차이즈는 일괄 제외하지 않는다. 대신 여행 가치 감점을 적용한다. 노포, 대전 향토음식, 지역 대표 메뉴, 본점, 전통시장 연계, 대전에서 시작한 브랜드 같은 신호가 있으면 감점을 상쇄할 수 있다. 자동 판정이 불확실한 후보는 제거하지 않고 `pending` 상태로 남긴다.

## 중복 판정

상호명은 HTML 태그, 법인 표기, 공백, 문장부호와 괄호 안의 단순 지역 표기를 제거해 비교한다. 주소는 `대전광역시` 같은 광역 표기와 불필요한 공백을 정리한다.

다음 중 하나면 기존 추천 목록과의 확정 중복으로 제외한다.

1. 정규화한 도로명 또는 지번 주소가 같고 정규화한 상호명이 같음
2. 상호명 유사도가 0.92 이상이고 기존 DB 좌표와 거리가 50m 이하

상호명은 같지만 주소와 좌표가 다른 지점은 별도 후보로 허용한다. 상호명 유사도는 높지만 좌표가 없거나 주소 비교가 불충분하면 제외하지 않고 `possible_duplicate=Y`로 표시한다.

같은 실행에서 수집된 후보도 같은 규칙으로 병합한다. 병합할 때 검색어와 노출 횟수는 누적한다.

## 점수와 정렬

점수는 100점 기준의 검토 우선순위이며 품질 보증이나 네이버 평점을 의미하지 않는다.

- 여러 검색어에서 반복 발견: 최대 20점
- `comment` 정렬에서 반복 발견: 최대 15점
- 음식점 세부 분류와 지역 메뉴 적합성: 최대 15점
- 블로그 결과량: 로그 스케일 최대 10점
- 최근 12개월 언급: 최대 15점
- 서로 다른 블로거 수: 최대 10점
- 노포, 향토음식, 본점, 전통시장, 대전 대표 메뉴 신호: 최대 15점
- 배달 중심 음식, 일반 체인, 메뉴 개성 부족 신호: 최대 30점 감점

각 구에서 `recommendation_score` 내림차순, `name` 오름차순으로 정렬한다. 점수가 양수인 후보 중 상위 100개를 내보내며, 통과 후보가 80개 미만이면 양수 후보 전부를 내보낸다.

## 오류 처리와 재실행

- 인증 실패와 잘못된 요청 같은 4xx 오류는 즉시 실패 처리한다. 단, 429는 재시도한다.
- 429와 5xx, 네트워크 오류는 지수 백오프로 최대 3회 재시도한다.
- 응답 필드가 누락되면 빈 값으로 정규화하되 이름이나 주소가 없으면 후보에서 제외한다.
- 실행 중 실패한 검색어와 HTTP 상태만 로그에 남기고 API 원문 응답은 저장하지 않는다.
- 출력은 임시 파일에 완성한 뒤 대상 CSV로 교체해 중간 실패로 기존 파일이 손상되지 않게 한다.
- 같은 API 응답과 기존 입력을 사용하면 정렬과 CSV 열 순서가 동일하도록 결정적으로 처리한다.

## 사용 방법

수집기는 기존 `.env`의 `NAVER_CLIENT_ID`와 `NAVER_CLIENT_SECRET`을 사용한다. 키 값은 로그, 예외 메시지, CSV에 출력하지 않는다.

기본 실행 명령은 다음과 같다.

```powershell
python scripts/collect_naver_restaurant_candidates.py
```

검증과 반복 실행을 위해 다음 옵션을 제공한다.

- `--district`: 한 구만 수집
- `--output`: 출력 파일 변경
- `--max-per-district`: 기본값 100
- `--skip-blog`: 지역검색과 중복 제거만 시험
- `--dry-run`: API를 호출하지 않고 설정과 입력 파일만 검증

## 테스트와 검증

네트워크 호출은 테스트에서 가짜 응답으로 대체한다.

- 상호명과 주소 정규화
- HTML 표시 태그 제거
- 좌표 거리와 상호 유사도를 결합한 기존 목록 중복 제거
- 다른 지점의 동일 브랜드 유지
- 배달형·카페 필터와 지역성 상쇄 규칙
- 블로그 최근 12개월 및 작성자 다양성 집계
- 점수 상한, 감점, 안정된 정렬
- CSV 열 순서, 한 줄 텍스트, UTF-8 출력
- 429와 5xx 재시도, 4xx 즉시 실패
- 구별 최대 수량과 부족 수량 처리

실제 API 스모크 테스트는 한 개 구와 낮은 최대 수량으로 실행한 뒤 전체 수집을 수행한다. 전체 결과에서는 구별 개수, 기존 목록 확정 중복 0건, 필수 필드 누락, 대상 구 밖 주소, CSV 파싱 성공 여부를 자동 보고한다.

## 이용 범위

네이버 API는 후보 탐색과 내부 검수 보조에 사용한다. 원문 응답과 블로그 본문은 축적하지 않으며 최종 추천 반영은 사람의 승인을 거친다. 운영 또는 외부 배포 전에 현재 네이버 API 이용약관이 애플리케이션의 저장·표시 방식과 맞는지 사용자가 확인해야 한다.

## 완료 기준

- 수집기 단위 테스트가 통과한다.
- 실제 API로 다섯 구 수집이 완료된다.
- 후보 CSV가 다섯 구를 포함하고 각 구에 최대 100개가 기록된다.
- 기존 추천 CSV와의 확정 중복이 없다.
- 자동 제외 사유와 불확실한 중복 표시가 검수 가능하게 남는다.
- API 키나 블로그 본문이 산출물과 로그에 포함되지 않는다.

# TAGO 버스 데이터 단일화 작업 내역 — 2026-07-14

## 1. 작업 목적

기존 `transport` 테이블에는 다음 두 원천의 버스 정류소가 혼합되어 있었다.

- 대전광역시 BIS 버스정류소 현황 API
- 국토교통부 TAGO 버스노선정보 API

두 원천은 서로 다른 정류소 ID 체계를 사용한다. 기존 구현은 정류소 이름이 같고 좌표가 30m 이내이면 TAGO 정류소를 BIS 정류소와 같은 정류소로 간주했다. 이 때문에 다음 문제가 있었다.

- 경로 그래프에서 BIS ID와 TAGO ID가 혼용됨
- 노선이 연결되지 않은 BIS 정류소 579개가 `transport`에 남음
- 여러 TAGO `nodeId`가 하나의 BIS 정류소로 합쳐짐
- TAGO 실시간 API 호출을 위해 원본 `nodeId`를 다시 복원해야 함
- 이름과 좌표를 이용한 추정 매칭 결과에 경로 정확도가 의존함

버스 노선 탐색과 실시간 조회의 기준 식별자를 일치시키기 위해 운영 버스 데이터를 TAGO 하나로 단일화했다.

## 2. 사용한 TAGO 데이터

- 도시코드: `25` — 대전광역시
- 노선 목록: `BusRouteInfoInqireService/getRouteNoList`
- 노선별 경유 정류소: `BusRouteInfoInqireService/getRouteAcctoThrghSttnList`
- 정류소 기본키: TAGO `nodeId`
- 노선 기본키: TAGO `routeId`
- 방향: TAGO `updowncd`
- 경유 순서: TAGO `nodeord`

검증에 사용한 저장 원천은 다음 파일이다.

```text
data/raw/tago_route_list_page*.json
data/raw/tago_route_stops_*.json
```

원천 수집 기준일은 2026-07-13이다. 2026-07-14에 TAGO 공식 노선 목록을 다시 조회한 결과 신설·폐지·메타정보 변경 노선은 없었다.

## 4. 변경 후 데이터

| 테이블 | 건수 | 기준 식별자 |
|---|---:|---|
| `transport` | 3,061 | TAGO `nodeId` |
| `bus_route` | 128 | TAGO `routeId` |
| `bus_route_stop` | 9,874 | `routeId + updowncd + nodeord` |

변경 후 상태는 다음과 같다.

- 모든 `transport.stop_id`가 TAGO 원본 `nodeId`와 일치
- `transport.source_api`는 전부 `tago`
- BIS 숫자형 정류소 ID 0개
- 노선이 비어 있는 정류소 0개
- 모든 경유 순서가 TAGO 원천과 일치
- `transport.routes`는 `bus_route_stop`과 `bus_route`에서 파생한 표시용 캐시

## 5. 스키마 변경

`transport`는 TAGO 단일 원천만 허용하도록 변경했다.

```sql
CREATE TABLE transport (
  stop_id    TEXT PRIMARY KEY,
  name       TEXT NOT NULL,
  type       TEXT NOT NULL DEFAULT 'bus',
  lat        REAL NOT NULL,
  lng        REAL NOT NULL,
  routes     TEXT,
  source_api TEXT NOT NULL DEFAULT 'tago'
                  CHECK (source_api = 'tago')
);
```

`bus_route_stop`에는 노선과 정류소 외래키를 추가했다.

```sql
CREATE TABLE bus_route_stop (
  route_id   TEXT NOT NULL
             REFERENCES bus_route(route_id) ON DELETE CASCADE,
  updowncd   INTEGER NOT NULL,
  node_order INTEGER NOT NULL,
  stop_id    TEXT NOT NULL REFERENCES transport(stop_id),
  PRIMARY KEY (route_id, updowncd, node_order)
);
```

## 6. 변경 파일

### `scripts/build_route_stops.py`

TAGO 원천 캐시를 검증하고 DB를 전체 스냅샷 방식으로 교체하도록 재작성했다.

- 기본 실행은 읽기 전용 dry-run
- 모든 노선과 정류소 캐시의 응답 코드 검사
- 페이지 `totalCount`와 실제 항목 수 비교
- 필수 필드 누락 검사
- `routeId + updowncd + nodeord` 중복 검사
- 노선 목록과 노선별 정류소 파일의 전체 커버리지 검사
- 검증 완료 후에만 단일 트랜잭션으로 테이블 교체
- 실제 반영 전 `db/backups`에 SQLite 백업 자동 생성
- 적재 후 외래키와 SQLite 무결성 검사

### `collectors/tago_bus_routes.py`

TAGO 원천 수집과 DB 반영을 분리했다.

- TAGO API 결과를 raw JSON으로만 저장
- 노선 목록 및 노선별 정류소 페이지 누락 검사
- 일부 노선 수집 실패 시 전체 작업 실패 처리
- DB는 직접 수정하지 않음

### `collectors/transport_bis.py` 삭제

TAGO 단일화 이후 다시 사용되지 않도록 대전시 BIS 수집기를 삭제했다.

- BIS API 인증키 설정(`DAEJEON_BIS_API_KEY`)도 `.env.example`에서 제거
- 기존 `data/raw/daejeon_bis_*.json`은 과거 수집 기록일 뿐 DB 빌드에는 사용하지 않음
- 운영 버스 데이터는 `collectors/tago_bus_routes.py`만 사용

### `collectors/common.py`

`upsert_transport()`가 `source_api='tago'`가 아닌 행을 거부하도록 변경했다.

### `db/schema.sql`

- `transport.source_api` 추가
- TAGO 단일 원천 `CHECK` 제약 추가
- 필수 컬럼 `NOT NULL` 강화
- `bus_route_stop` 외래키 추가

### `.gitignore`

마이그레이션 전 자동 생성되는 로컬 DB 백업을 Git에서 제외했다.

```text
db/backups/
```

## 7. 실제 반영 절차

먼저 기존 DB 복제본에서 전체 마이그레이션을 시험한 후 실제 DB에 반영했다.

```powershell
# 원천 캐시 및 스냅샷 검증 — DB 변경 없음
python scripts/build_route_stops.py

# 기존 DB 자동 백업 후 TAGO 스냅샷 반영
python scripts/build_route_stops.py --apply
```

마이그레이션 후 사용하지 않는 SQLite 페이지를 정리하기 위해 `VACUUM`도 수행했다.

## 10. 앞으로의 정기 갱신 절차

```powershell
# 1. 최신 TAGO 노선 및 경유 정류소 수집
python collectors/tago_bus_routes.py

# 2. 전체 캐시 검증
python scripts/build_route_stops.py

# 3. 검증 결과 확인 후 DB 반영
python scripts/build_route_stops.py --apply
```

운영 시 주의사항:

- TAGO 수집 중 일부 노선이 실패하면 DB 반영을 진행하지 않는다.
- `build_route_stops.py` dry-run에서 건수나 페이지 오류가 나오면 기존 DB를 유지한다.
- TAGO `nodeId`가 다르면 이름과 좌표가 가까워도 별도 방향 정류소로 유지한다.
- 대전시 BIS 수집기는 삭제됐으며 기존 BIS raw 파일은 운영 경로 그래프에 사용하지 않는다.
- 실시간 버스 위치 및 도착 정보는 정적 DB에 저장하지 않고 응답 시점에 조회한다.
